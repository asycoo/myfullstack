# Next.js + Prisma + PostgreSQL 全栈入门学习规划（目标：跑通一个可部署项目）

面向人群：**前端开发**，后端几乎从 0 开始  
目标：用 **Next.js（App Router）+ Prisma + PostgreSQL（Docker）** 做出一个**可登录、可增删改查、可部署**的最小全栈项目，并在过程中系统补齐后端知识。

---

## 你正在做的项目（建议范围）
做一个「迷你博客/笔记系统」（最小但完整）。

- **核心功能**
  - 注册 / 登录 / 退出
  - 文章（或笔记）列表 / 详情 / 新增 / 编辑 / 删除
  - 权限：必须登录；只有作者能编辑/删除自己的内容
- **数据表（从小到大扩展）**
  - `User`
  - `Post`（或 `Note`）
  - 后续再加：`Tag`、`Comment`、软删除、审计字段等

---

## 环境与目录（你当前仓库的推荐结构）

- `docker-compose.yml` 放在项目根目录：负责基础设施（PostgreSQL）
- `web/` 放 Next.js 项目：负责前后端应用代码
- `DOC/` 放学习与记录文档：沉淀过程与踩坑

这样做的意义：**基础设施与应用代码分离**，不容易互相覆盖；也更接近真实团队工程结构。

## 阶段 0（第 0 周，1–2 天）：环境一次性搭好

### 你要完成什么
- [ ] Docker Desktop 正常运行（能 pull、能 run 镜像）
- [ ] PostgreSQL 用 `docker compose` 启动并长期后台运行
- [ ] Next.js 项目能启动并在浏览器访问

### 每一步的意义
- **Docker**：让数据库环境可复制、可清理、不污染系统；后续部署/CI 思路一致
- **PostgreSQL**：最常用的关系型数据库之一，生态成熟
- **Next.js**：同一个工程里同时练到「后端接口 + 前端页面」

## 阶段 1（第 1 周）：Next.js 后端入门（先会写 API）

### 学习重点
- **App Router**：`web/src/app` 下的路由组织方式
- **Route Handlers**：`web/src/app/api/**/route.ts` 写接口
- **HTTP 基础**：方法、状态码、请求体、响应体、Header

### 你要做的功能（最小闭环）
- [ ] `GET /api/health`：返回 `{ ok: true, time: ... }`
- [ ] 前端页面请求 `health` 并展示结果（用 `fetch`）

### 这一步的意义
先把“**页面 → 调接口 → 得到响应**”打通，建立你对全栈的基本直觉：
前端发什么、后端收什么、后端回什么、前端怎么处理错误。

---

## 阶段 2（第 2 周）：Prisma + 数据库（从建模到迁移）

### 学习重点
- `schema.prisma`：用“代码”描述数据库结构（模型、字段、约束、关系）
- `prisma migrate`：把 schema 变更**安全地**同步到数据库（迁移历史可追踪）
- Prisma Client：用类型安全方式做 CRUD

### 你要做的功能
- [ ] 建模 `User`、`Post`
- [ ] 完成 Post CRUD（含分页）
  - `GET /api/posts?page=1&pageSize=10`
  - `GET /api/posts/:id`
  - `POST /api/posts`
  - `PATCH /api/posts/:id`
  - `DELETE /api/posts/:id`

### 这一步的意义
你会第一次体会到“**数据库结构变更**”与“**代码发布**”之间的关系：

- **建模（schema）**：决定数据怎么存、怎么关联、怎么约束
- **迁移（migrate）**：把这份变化变成可回溯的历史，团队协作/上线都靠它
- **CRUD**：把业务需求落到可用接口上（列表、详情、编辑、删除）

---

## 阶段 3（第 3 周）：登录与鉴权（让项目“像真的”）

### 学习重点
- 密码存储：**只存哈希，不存明文**（bcrypt/argon2）
- 登录态：先用 **Cookie（HttpOnly）+ 会话** 的思路（实现更直观）
- 权限控制：接口层面做“是否登录”“是否作者”

### 你要做的功能
- [ ] `POST /api/auth/register` 注册
- [ ] `POST /api/auth/login` 登录（下发 HttpOnly Cookie）
- [ ] `POST /api/auth/logout` 退出（清 cookie）
- [ ] Post 写接口加保护：未登录返回 401；非作者返回 403

### 这一步的意义
- 你会掌握后端最核心的能力之一：**身份识别与权限控制**
- 你会理解为什么前端“看起来只是多了一层登录页”，后端却要做很多安全细节（cookie、hash、权限）

---

## 阶段 4（第 4 周）：输入校验、错误处理、工程化（从“能跑”到“能用”）

### 学习重点
- 参数校验：用 **Zod** 校验 query/body（避免脏数据进库）
- 统一返回结构：前端更好处理错误与提示
- 日志：能定位“哪次请求、哪个用户、哪个接口”出了问题

### 你要做的功能
- [ ] 所有写接口加 Zod 校验（创建/更新/分页）
- [ ] API 统一返回格式（建议）
  - 成功：`{ data: ..., error: null }`
  - 失败：`{ data: null, error: { code, message } }`
- [ ] 统一错误映射：校验失败 400、未登录 401、无权限 403、资源不存在 404

### 这一步的意义
- **校验**：把 bug 挡在入口，避免数据库里出现“修不完的脏数据”
- **统一错误**：前端不再到处 `try/catch` 拼字符串，体验一致
- **日志**：以后你调试线上问题，靠的就是它

---

## 阶段 5（第 5–6 周）：部署上线（全栈闭环完成）

### 学习重点
- 环境变量：本地/线上不同配置如何管理（例如 `DATABASE_URL`）
- 数据库迁移上线策略：线上部署时 migrate 怎么跑
- 部署平台：先选一个跑通（例如 Vercel + 托管 Postgres）

### 你要做的功能
- [ ] 选择一个部署方案并跑通
- [ ] 线上数据库可用、迁移可执行、应用可访问
- [ ] 写 `README`：本地启动、迁移、部署步骤

### 这一步的意义
- 你将完成“从 0 到可访问产品”的最后一公里：**别人能打开你的链接并使用**
- 你会理解后端真正的难点之一：**环境差异与发布流程**

## 每周节奏建议（最稳的学习方式）
- **2 天**：读概念 + 跟官方文档跑最小例子（只学“够用”的）
- **3 天**：只做项目（把接口做完整，页面做最小可用）
- **1 天**：复盘与补强（错误处理、校验、重构、小结）
- **1 天**：休息或扩展（性能/缓存/队列/文件上传等）
